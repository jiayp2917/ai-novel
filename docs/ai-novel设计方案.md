# ai-novel 设计方案

本文描述 `ai-novel` 的产品定位、核心边界、信息架构和长期蓝图。它回答“项目最终应长成什么样，以及哪些设计原则不能被破坏”。

## 1. 产品定位

`ai-novel` 是本地优先、人工可控、AI 辅助的通用长篇小说生产工作台。它不绑定单一作品，也不把某本书的人物、题材、世界观、章节状态或风格偏好写入全局规则。

短期目标：

- 支撑 20-30 万字作品从构思、设定、大纲、章纲、正文候选、审核、修订到发布的完整闭环。
- 让作者即使在模型不可用时，也能完成正文编辑、版本保存、diff 和人工发布。

长期目标：

- 支撑百万字级别作品的分层记忆、人物状态、伏笔生命周期、时间线、风格学习、成本控制和发布审计。
- 降低长篇创作中的设定漂移、人物跑偏、伏笔遗忘、重复桥段和 AI 味。

## 2. 目标用户

- 需要长期稳定产出的网文作者。
- 有构思但希望 AI 辅助补设定、拆章纲、生成初稿和做一致性检查的作者。
- 喜欢手写正文，但需要 AI 做扩写、润色、校验、伏笔检查和风格归纳的作者。
- 同时管理多题材、多作品、多版本内容的创作者。

## 3. 核心原则

1. 作者拥有最终控制权。
2. AI 输出只能先成为 proposal、artifact 或候选版本。
3. 正文写回必须经过 review、diff、人工确认、backup 和 publish gate。
4. 设定、大纲、章纲、作品档案、长期记忆和风格规则必须 proposal-first。
5. 单章是正文生成、审核、修订、发布和统计的最小单位。
6. 自动流水线默认 dry-run，不做无人值守全书发布。
7. 不同作品的工作区、运行态、记忆、风格和模型统计必须隔离。
8. 密钥、个人配置、真实作品内容和运行态产物不得进入公开仓库。
9. 前端普通用户可见文案中文优先，除产品名、供应商/模型名、技术字段、路径、命令、API、JSON、调试信息、用户原文和第三方系统提示外，不新增英文界面文案。
10. 所有状态变更、AI 调用、运行态写入和正文写回接口默认只允许本机访问；远程或反向代理部署必须使用显式 token。
11. 视觉设计以“作者写作台”为主语，不以 AI Dashboard 为主语；主题材质只能辅助氛围，不能牺牲正文阅读、按钮稳定性和发布门辨识度。

## 4. 信息架构

前端保持六个入口。

| 入口 | 定位 | 关键边界 |
| --- | --- | --- |
| 首页 | 当前作品概览和快捷入口 | 不直接编辑正文，不直接发布 |
| 写作 | 作者日常正文编辑主界面 | 不承载模型排错细节 |
| AI 素材库 | 设定、大纲、章纲和作品资料 proposal | 不提供正文写回入口 |
| AI 工作台 | 草稿审核、修订、diff 和写回确认 | 不替代人工写作页 |
| 自动流水线 | 批量章节候选生产和状态观察 | 默认 dry-run，不绕过发布门 |
| 设置/模型 | 工作区、模型、质量、用量和高级日志 | 模型档案池和角色分配，密钥只显示状态 |

界面文案面向写作者而不是工程排错人员。普通按钮、标题、空状态、表单提示和错误反馈应使用中文；必要英文应收敛在模型名、路径、环境变量、原始日志和高级调试区域。

界面提供三套主题：主题1“清风微动”、主题2“星空鎏金”、主题3“白丝质感”。三套主题共享同一信息架构和操作路径，只改变材质、色彩、对比和氛围；主题图片必须是无文字、无人物、无 UI 的抽象背景，避免把应用做成宣传页或通用 AI 模板。

主流程串联：

```text
首页
-> 选择或创建工作区
-> 扫描素材
-> 写作 / AI 素材库 / AI 工作台 / 自动流水线
-> diff
-> 人工确认发布
```

## 5. 工作区与资料边界

每个作品应有独立工作区；具体支持的目录结构由使用手册维护。设计层只规定边界：系统仓库放通用能力，小说工作区放作品资料、作品级 skill、运行态和审计记录。

系统仓库保存通用能力：

- 后端、前端、配置模板、通用 skills、测试、文档。
- 不保存真实作品正文、设定、章纲、运行态、个人密钥或作品级规则。

作品档案建议包含：

- 书名、题材、目标字数、目标读者、叙事人称、整体基调。
- 核心卖点、主角驱动、主要角色、世界规则摘要。
- 禁写项、卷/章规划、当前生产阶段。

作品档案也必须先通过 proposal 补全，用户确认后才成为正式资料。

## 6. AI 主导生产流

标准流程：

```text
作品构思
-> 作品档案 proposal
-> 设定 proposal
-> 总纲 proposal
-> 卷纲 proposal
-> 逐章章纲 proposal
-> 单章写作卡 proposal
-> 用户确认写作卡
-> 正文 artifact
-> reviewer 审核
-> fixer 修订
-> 复审
-> diff
-> 人工确认
-> backup
-> publish gate 写回
-> 扫描
-> 记忆/风格 proposal
```

角色职责：

- writer：只生成正文候选。
- reviewer：只审核，不修文、不写文。
- fixer：只修复 reviewer 授权且证据充分的问题。
- outliner：生成作品档案、设定、大纲、章纲和写作卡相关 proposal。
- memory/long_context：只基于已确认资料和已发布正文构建上下文。
- arbiter：做高风险辅助判断，不绕过发布门。

缺失信息应优先结构化问询。用户可以回答、跳过并让 AI 生成建议 proposal，或标记为暂不需要。系统不得为了推进流程而静默编造 canonical 资料。

## 7. 人工主导编辑流

写作页首先是文档编辑器，不依赖模型可用性。

基础能力：

- 新建章节、编辑正文、保存版本。
- 搜索、批注、版本历史、diff。
- 发布确认、备份提示、可隐藏侧栏。
- 后续补齐复制章节、软删除和恢复、全屏写作。

AI 选区能力：

- 扩写、润色、降低 AI 味。
- 错字校验、语病校验、格式校验。
- 一致性检查、逻辑 bug 检查、伏笔检查、人物口吻检查。

所有 AI 选区结果先生成候选。应用到编辑稿后也只是未发布版本，不能直接成为正式正文。

## 8. 单章写作卡

单章写作卡是稳定正文生产的核心输入，用于减少随机漂移和 token 浪费。

写作卡至少包含：

- 章节编号、章节标题、所属卷。
- 本章目标、起始状态、结束状态。
- 场景、出场人物、人物状态变化。
- 冲突或机制、常规推进、主角行动。
- 情绪点、爽点或悬念点。
- 信息增量、伏笔埋设或回收、结尾钩子。
- 禁写项、字数范围、风格摘要。
- 依赖设定和依赖记忆。

确认后的写作卡应足够具体，可以直接交给 writer 生成单章正文候选。写作卡必须进入 context report，记录 generation mode、skill、memory 和关键上下文来源。

## 9. 生成模式

正文和 AI 辅助编辑支持三类模式：

| 模式 | 目标 | 适用场景 |
| --- | --- | --- |
| 稳定省钱 | 输出稳定、成本低 | 日常章节默认模式 |
| 质量优先 | 降低关键章节风险 | 开篇、高潮、转折、结尾 |
| 速度优先 | 快速出草稿 | 低风险过渡章或人工主审 |

生成模式必须记录到 artifact metadata、model call 关联记录或 context report。稳定模式不应默认强制重新生成；相同输入允许复用缓存。

## 10. 记忆与风格学习

百万字项目不能依赖全文 prompt。writer 推荐上下文顺序：

```text
确认后的单章写作卡
-> 当前卷摘要
-> 最近 3-5 章摘要
-> 相关人物状态卡
-> 相关设定条目
-> 相关伏笔
-> 作品级短风格规则
-> 禁写项
```

记忆分层：

- 作品摘要：来自已确认作品档案和总纲。
- 卷摘要：来自已确认卷纲和已发布章节。
- 章节摘要：来自已发布正文。
- 人物状态卡：来自已发布正文和用户确认资料。
- 伏笔卡：记录埋设、推进、回收、废弃。
- 风格规则：来自用户确认的 style proposal。

风格学习流程：

```text
AI artifact
-> 用户修改并发布
-> diff
-> 风格差异报告
-> style proposal
-> 用户确认
-> 作品级风格规则版本
```

限制：

- 不从未发布草稿学习。
- 不复制长段正文进 skill。
- 不自动更新全局风格。
- 风格规则按作品隔离、版本化、可回滚。

## 11. 发布门设计

人工正文发布可以不经过 AI review，但必须查看 diff、确认发布、生成 backup、校验 hash、写入源文件并记录 publish decision。AI 正文发布必须先经过 artifact、review、必要修订、复审、diff 和人工确认。

禁止路径：

- AI 直接覆盖正文。
- review 未通过直接发布。
- 未生成 diff 发布。
- 未备份发布。
- 自动流水线默认真实写回。
- 缺少基础绑定的 artifact 进入 publish。

## 12. 自动流水线

自动流水线定位为辅助排队和观察，不是自动出版系统。

理想能力：

- 创建 dry-run 任务。
- 展示每章状态、当前步骤、失败原因和下一步动作。
- 支持合法状态下的 pause、resume、cancel、retry。
- 生成运行报告。
- 保留 artifact、review、diff 和事件记录。

安全默认：

- 默认只生成草稿，不写回正文。
- 不提供普通用户一键批量真实发布入口。
- 真实发布必须回到单章发布门。

## 13. 交付形态

阶段 A：浏览器访问型安装包。

目标体验：

```text
用户安装 ai-novel
-> 点击桌面图标
-> launcher 启动本机后端
-> 初始化数据库和用户数据目录
-> 托管或启动前端静态页面
-> 自动打开本机网页
-> 用户完成写作、AI 生成、审核、diff 和发布
```

阶段 A 必须做到：

- 不要求普通用户安装 Python、Node.js 或手动运行迁移。
- 用户数据目录和应用安装目录分离。
- 首次启动向导支持选择或创建作品工作区。
- 设置页支持保存模型配置状态，不要求用户手写配置文件。
- 模型不可用时，人工写作和发布手写版本仍可用。

阶段 B：桌面窗口型安装包。

在阶段 A 稳定后，再考虑 Electron 或 Tauri 桌面窗口、应用菜单、最近作品、日志导出、自动更新、崩溃恢复和后台进程守护。

当前不优先做纯云端 SaaS。小说正文、设定、大纲、模型密钥和运行态审计具有强隐私属性，现阶段本地优先更适合验证写作闭环。

## 14. 模型治理

模型配置采用“模型档案池 + 角色分配”：

- 模型档案描述供应商、模型、接口地址、密钥环境变量、输出上限和 JSON 支持能力。
- writer、reviewer、fixer、quick_fix、outliner、structural_fix、memory、long_context 和 arbiter 可以分配不同档案。
- 内置档案只读，自定义档案可编辑；被角色使用的档案不能删除。
- 密钥只保存和显示状态，不回显明文；高级排错区可以显示 provider/model、调用状态和脱敏错误。
- 这不是完整供应商管理平台，短期只服务小说写作角色路由和排错。

## 15. 外部实践抽象原则

后续可以从公开写作工具、Agent 工作流和本地优先应用中吸收抽象经验，但只沉淀通用原则，不复制专有 prompt、长段方法论原文或其它项目的私有内容。

可吸收的方向：

- 把写作流程拆成作品档案、设定、大纲、章纲、写作卡、正文候选、审核和修订。
- 为对标、拆文和风格差异建立资产库，但只保存抽象结论和短规则。
- 用多角色分工提高可观察性，避免 writer 同时承担审核和修复。
- 用蓝图、概念对话、多版本候选、RAG、摘要闭环和 Docker/安装体验改善普通用户入口。

禁止照搬：

- 不复制专有 prompt 或大段方法论原文。
- 不把某一项目的目录结构、世界观假设或交互流程硬套进本项目。
- 不绕过 `ai-novel` 已有的 proposal/artifact first、发布门、dry-run 默认和作品隔离原则。

## 16. 长篇扩展验收

百万字扩展必须具备：

- 分卷摘要、章节摘要、人物状态卡、伏笔生命周期和时间线。
- 地点、组织关系和设定变更影响分析。
- 重复桥段检测、人物口吻漂移检测和卷末复盘。
- token 成本、上下文命中率和发布失败恢复统计。

验收场景：

- 1000 章模拟数据下，素材扫描和上下文构建仍可用。
- 生成后期章节时不读取全书正文。
- 能检索到与当前写作卡相关的人物、伏笔、设定和最近章节摘要。
- 能指出未回收伏笔和人物状态冲突。

## 16. 后续决策规则

后续任何新功能都应先回答：

1. 是否会绕过 proposal 或 publish gate？
2. 是否会把单一作品规则写进全局？
3. 是否能在模型不可用时保持人工编辑主流程可用？
4. 是否能被测试、审计和回滚？

若任一答案不清楚，先做文档和 sandbox 验证，不直接进入真实作品写回。
