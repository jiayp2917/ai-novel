# 写作工作流迭代需求清单

本文把参考项目中的可借鉴能力转化为 `ai-novel` 的后续迭代需求。需求按 P0、P1、P2 排序，优先服务“本地优先、作品隔离、人工确认、可追踪写回”的长篇小说编辑器目标。

## 1. 迭代原则

可借鉴点：

- 写作能力应围绕作者真实流程组织：构思、设定、大纲、章纲、正文、审查、修订、发布、记忆更新。
- AI 输出应先成为 proposal 或 artifact，用户确认后才进入正式资料或正式正文。
- 单章是最小生产单位，批量流水线只能排队多个单章任务。
- 已发布正文和已确认资料才可以进入长期记忆、风格规则和检索上下文。

不建议照搬：

- 不把参考项目的完整 prompt、skill、示例小说或私有运行方式并入仓库。
- 不把自动流水线描述为无人值守全书发布。
- 不把云端账号、复杂 Agent 文件体系或向量数据库作为早期必需条件。

`ai-novel` 落地建议：

- 所有新功能都要明确输入来源、输出类型、确认边界和审计记录。
- UI 只把作者需要操作的内容放在主路径，工程细节放到高级区。
- 每个功能优先设计可测的最小闭环，再扩展批量和质量模式。

## 2. P0：先打通稳定写作闭环

### 2.1 作品档案补全

可借鉴点：

- 用结构化问询把零散构思转成作品档案。
- 作品档案作为后续设定、大纲、章纲和写作卡的根输入。

不建议照搬：

- 不直接生成并覆盖正式作品档案。
- 不要求用户一次填完所有字段。

`ai-novel` 落地建议：

- 增加作品档案缺字段检测：题材、目标读者、核心卖点、主角驱动、世界规则、目标字数、禁写项。
- 缺字段时生成补全 proposal 或弹出结构化问询。
- 用户可逐项采纳、跳过或手工填写。

验收标准：

- 缺少作品档案时不阻断人工编辑。
- AI 生成的作品档案补全只进入 proposal。
- 采纳后保留事件记录和来源 artifact。

### 2.2 单章写作卡

可借鉴点：

- 正文生成前先锁定本章目标，减少模型漂移。
- 写作卡包含情绪目标、核心事件、出场人物、冲突、钩子、禁写项和字数范围。

不建议照搬：

- 不把写作卡做成复杂方法论文本。
- 不在未确认写作卡的情况下默认进入稳定正文生成。

`ai-novel` 落地建议：

- 从章纲生成写作卡 proposal。
- 用户确认后，writer 在稳定模式下优先读取写作卡。
- artifact metadata 记录写作卡 ID、生成模式、上下文来源和模型角色。

验收标准：

- 同一章有可查看的写作卡版本。
- writer context report 能显示是否使用写作卡。
- 未确认写作卡不会被当作正式输入。

### 2.3 正文候选、审查、修订、发布门

可借鉴点：

- 正文候选与正式正文分离。
- reviewer 找问题，fixer 按证据修复，用户最终确认发布。

不建议照搬：

- 不让 reviewer 直接改正文。
- 不让 fixer 修复无证据或越界问题。
- 不让候选选择绕过 diff、backup 和 publish decision。

`ai-novel` 落地建议：

- AI 工作台主路径固定为：生成候选 -> review -> fixer -> 复审 -> diff -> 用户确认发布。
- 人工正文版本可不强制 AI review，但仍需 diff、backup 和发布记录。
- 强制发布必须记录理由，不能跳过备份。

验收标准：

- AI 候选未通过 review 时不能直接发布。
- 发布记录可追溯 artifact、review、diff、backup 和用户确认。
- 原文件 hash stale 时阻断发布并提示重新扫描或合并。

### 2.4 发布后摘要与短记忆

可借鉴点：

- 每章发布后提取真实摘要，为下一章提供上下文。
- 摘要来自已发布正文，不来自未确认章纲。

不建议照搬：

- 不让未发布草稿进入长期记忆。
- 不在早期强依赖向量库。

`ai-novel` 落地建议：

- 发布成功后触发 summary/memory 任务。
- 先生成章节摘要、最近章节摘要和角色状态 proposal。
- 用户确认后再进入长期资料；短上下文可用于下一章候选，但要标记来源。

验收标准：

- 已发布章节能生成摘要 artifact。
- 下一章 context report 包含最近章节摘要来源。
- 未发布 artifact 不进入 memory。

## 3. P1：提升长篇生产质量

### 3.1 导入已有作品

可借鉴点：

- 从已有章节反向整理设定、角色、剧情线、时间线、伏笔和文风摘要。
- 导入目标是可续写工程，而不只是生成一份分析报告。

不建议照搬：

- 不复制参考项目的拆文管道实现。
- 不把导入分析结果直接覆盖正式设定。
- 不把真实原文放进系统仓库。

`ai-novel` 落地建议：

- 新增“导入已有作品”向导，支持选择工作区内正文目录或单个文本文件。
- 输出为导入报告、设定 proposal、章纲 proposal、角色状态 proposal、伏笔/时间线 proposal。
- 原文保留在用户作品工作区，系统仓库只保存代码和通用文档。

验收标准：

- 导入后原正文 hash 不变。
- 分析结果全部是 proposal 或只读 artifact。
- 用户可选择采纳单个角色、单条设定或单章章纲。

### 3.2 对标/拆文资产库

可借鉴点：

- 对标作品可拆成结构、节奏、角色功能位和文风摘要，作为写作参考。
- 对标资产和当前作品正式资料分离。

不建议照搬：

- 不把对标原文作为默认 prompt 内容。
- 不让对标设定污染当前作品 canonical 资料。

`ai-novel` 落地建议：

- 增加“参考资产”类型：对标报告、节奏摘要、角色功能位、文风摘要。
- 写作卡和 writer 上下文只能引用摘要级信息，不能默认引用大段原文。
- UI 中明确标识“参考资产”和“正式资料”。

验收标准：

- context report 能区分参考资产和正式作品资料。
- 参考资产未采纳前不进入正式设定。
- 删除参考资产不会影响正式正文和正式设定。

### 3.3 角色状态、伏笔和时间线

可借鉴点：

- 长篇需要持续追踪角色状态、伏笔生命周期和故事时间线。
- 每章写完后立刻更新追踪资料，比事后补救更可靠。

不建议照搬：

- 不要求每个路人角色都建档。
- 不自动生成大量未确认状态，避免污染后续上下文。

`ai-novel` 落地建议：

- 为角色状态、伏笔、时间线建立 proposal 类型。
- 发布后由 memory 角色生成更新建议，用户可采纳或忽略。
- 写作卡生成时读取已确认的相关状态和待处理伏笔。

验收标准：

- 同一角色状态变更可追溯来源章节。
- 伏笔有状态：埋下、推进、回收、废弃。
- 时间线冲突进入 reviewer finding，而不是静默忽略。

### 3.4 多版本候选

可借鉴点：

- 同一章生成多个候选，用户可比较选择。
- 多版本适合关键章节、开篇和高潮。

不建议照搬：

- 不把多候选作为默认，避免成本失控。
- 不让用户选择候选后直接覆盖正文。

`ai-novel` 落地建议：

- 稳定模式默认单候选。
- 质量优先模式允许 2-3 个候选。
- UI 展示候选差异、review 结论、字数和上下文符合度。

验收标准：

- 每个候选有独立 artifact 和 model call。
- 被选候选进入正文版本，不直接写源文件。
- 未选候选保留审计但不进入 memory。

### 3.5 结构化审查报告

可借鉴点：

- findings 应包含严重度、类别、位置、证据和修复建议。
- 审查可以按 full、lean、solo 或质量等级执行。

不建议照搬：

- 不依赖外部 Agent 文件存在才可审查。
- 不把审查报告写成不可机器消费的长评。

`ai-novel` 落地建议：

- 统一 reviewer 输出 schema。
- fixer 只消费结构化 findings。
- UI 按严重度和类别筛选问题。

验收标准：

- reviewer 输出能被 fixer 自动读取。
- 没有证据的位置不能进入自动修复。
- S1/S2 问题默认阻断 AI 候选发布。

## 4. P2：改善使用与部署体验

### 4.1 作者首页

可借鉴点：

- 普通作者需要看到“下一步做什么”，不是任务队列细节。

不建议照搬：

- 不把首页做成营销页。
- 不在首页暴露模型 raw payload。

`ai-novel` 落地建议：

- 首页展示继续写作、待确认提案、待审查草稿、最近发布章节、工作区健康状态。
- 高级诊断入口折叠展示。

验收标准：

- 新用户打开后能直接选择或创建作品工作区。
- 模型不可用时人工编辑入口仍可用。

### 4.2 普通用户启动路径

可借鉴点：

- Docker 或桌面启动器能降低试用门槛。
- 常见错误应有面向作者的解释。

不建议照搬：

- 不把开发者启动命令作为普通作者唯一入口。
- 不把密钥存储到前端或提交到仓库。

`ai-novel` 落地建议：

- 用户手册拆分为普通作者、开发者、运维三条路径。
- 设置页提供模型连接测试、工作区扫描测试和发布门自检。
- 后续评估 Docker 或桌面启动器，但数据目录必须和源码目录分离。

验收标准：

- README 能在 5 分钟内指导开发者启动项目。
- 用户手册能解释模型失败、扫描失败、发布失败和备份恢复。
- 文档不包含本地私有路径或密钥。

### 4.3 RAG 与长篇检索

可借鉴点：

- 长篇不能依赖全文 prompt，应使用摘要、分块和检索。

不建议照搬：

- 不在 P0 阶段引入必须部署的复杂向量数据库。
- 不把草稿、失败候选和未确认 proposal 纳入检索源。

`ai-novel` 落地建议：

- 第一阶段使用文件摘要和轻量索引。
- 第二阶段再加入向量检索，并限制来源为已发布正文和已确认资料。
- 检索失败时降级到写作卡、最近摘要和作品档案。

验收标准：

- 上下文构建耗时和来源数量可观测。
- 检索结果在 context report 中可追溯。
- 删除或更新章节后不会读取过期摘要。

## 5. 下一步实施顺序

1. P0-1：作品档案补全 proposal。
2. P0-2：单章写作卡生成、确认和 writer 接入。
3. P0-3：结构化 reviewer findings 与 fixer 消费。
4. P0-4：发布后章节摘要和短记忆任务。
5. P1-1：导入已有作品的只读分析和 proposal 输出。
6. P1-2：对标/参考资产库和 context report 来源标记。
7. P1-3：角色状态、伏笔和时间线 proposal。
8. P1-4：多候选质量模式。
9. P2-1：作者首页和普通用户启动路径。
10. P2-2：轻量检索到向量检索的分阶段实现。

## 6. 验收总表

| 能力 | 最小验收 |
| --- | --- |
| 作品档案补全 | 缺字段生成 proposal，采纳后可追溯 |
| 写作卡 | 已确认写作卡进入 writer context report |
| 正文候选 | AI 候选必须经 review/diff/publish gate |
| 发布后摘要 | 只从已发布正文生成，下一章可读取 |
| 导入已有作品 | 原文不变，分析结果不直接覆盖正式资料 |
| 对标资产 | 参考资产与正式资料隔离 |
| 角色/伏笔/时间线 | 发布后生成 proposal，用户确认后生效 |
| 多版本候选 | 质量模式可生成多个候选，未选候选不入 memory |
| 作者首页 | 模型不可用时仍可人工写作 |
| 检索上下文 | 来源可追溯，未确认草稿不参与 |
