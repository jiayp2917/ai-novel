# 参考项目写作与使用借鉴分析

本文用于整理两个参考项目在写作方法、产品使用和工程组织上的可借鉴点，并转化为 `ai-novel` 后续迭代依据。本文只记录公开可讨论的产品和工程观察，不复制参考项目的大段提示词、正文样例或本地配置。

## 1. 分析结论

`oh-story` 更偏向写作方法论和工程化技能库，价值在于把网文创作拆成扫榜、拆文、设定、大纲、正文、审查、去 AI 味等明确阶段，并用文件系统承载长期上下文。

`arboris` 更偏向产品化小说工作台，价值在于围绕普通作者的使用路径设计概念对话、蓝图、章节生成、多版本候选、评审、摘要提取、RAG 检索和部署体验。

`ai-novel` 当前适合吸收两者的长处，但应保持自己的核心边界：本地优先、作品隔离、proposal/artifact first、正文发布门、自动流水线默认 dry-run。

## 2. oh-story 可借鉴点

### 2.1 写作流程分层

可借鉴点：

- 将创作拆成“市场/题材判断 -> 对标拆文 -> 作品设定 -> 卷纲/章纲 -> 正文 -> 审查 -> 精修”的连续流程。
- 对长篇写作强调“每章先有细纲”，避免直接让 AI 从模糊大纲生成正文。
- 用“情绪目标、章首钩子、爽点、章尾钩子、字数目标”约束单章，不让正文生成变成随机抽样。
- 对日更续写、大修、开书三种场景做路由，减少用户每次都重新描述上下文的成本。

不建议照搬：

- 不应把参考项目的 skill 指令原样并入 `ai-novel`。这些指令和运行机制绑定其宿主工具，直接搬运会造成重复规则、版权风险和维护困难。
- 不应把单一网文平台或单一题材规则写死为全局默认。`ai-novel` 是通用长篇小说编辑器，题材、平台和风格应保持作品级隔离。

`ai-novel` 落地建议：

- 在 AI 素材库中增加“创作阶段”视图：作品档案、设定、总纲、卷纲、章纲、写作卡、正文 artifact、review、风格 proposal。
- 将单章写作卡扩展为稳定输入，至少包含情绪目标、核心事件、冲突、出场人物、钩子、禁写项、字数范围和依赖资料。
- 对“开书、续写、大修”建立显式入口，用户选择后系统自动加载对应资料，不靠用户在对话里反复说明。

### 2.2 对标与拆文资产

可借鉴点：

- 把对标作品拆成角色、剧情、设定、章节摘要、文风摘要等结构化资产。
- 写作时只召回与当前章节有关的少量材料，而不是把整本对标或整本作品塞入 prompt。
- 通过“对标资产”和“当前作品资产”分离，避免参考内容污染正式设定。

不建议照搬：

- 不应把对标原文作为默认上下文。对标材料应只用于结构、节奏、风格特征提炼，不能鼓励贴近原文字句。
- 不应要求普通用户先理解复杂命令式 skill 体系。`ai-novel` 应通过界面引导完成“导入、分析、选择对标、生成建议”。

`ai-novel` 落地建议：

- 增加“参考/对标资产”类型，和正式设定、正式大纲分开存储。
- 对导入作品生成只读分析报告，产物先进入 proposal，不自动写入作品设定。
- 上下文构建器增加“当前章节相关对标摘要”输入槽，并在 context report 中记录来源。

### 2.3 Agent 分工与审查

可借鉴点：

- 将高层结构、角色设计、正文写作、一致性检查、资料研究、上下文查询、章节提取拆成不同职责。
- 审查流程强调“找问题”，并要求输出可定位、可修复的问题，而不是笼统评价质量。
- 支持 full、lean、solo 等不同审查强度，便于在成本和质量之间切换。

不建议照搬：

- 不应依赖某个外部 Agent 文件体系作为 `ai-novel` 的运行前提。
- 不应让多个 Agent 直接拥有写回权限。所有 Agent 输出仍应进入 artifact、review 或 proposal。

`ai-novel` 落地建议：

- 在现有 AI 角色上明确职责边界：writer 只产候选，reviewer 只审查，fixer 只按证据修复，memory 只处理已确认资料，arbiter 只辅助高风险裁决。
- 审查报告统一 schema，包含严重度、类别、证据、位置和建议修复方向。
- 给自动流水线增加审查强度选项：稳定、省钱、质量优先、快速草稿。

### 2.4 去 AI 味与风格学习

可借鉴点：

- 将 AI 腔、套话、总结体、对话说明书化等问题作为独立质量维度。
- 将标点、句长、段落密度、对话潜台词作为可检查项。
- 写作完成后更新状态、伏笔、时间线和角色状态，而不是只保存正文。

不建议照搬：

- 不应把“去 AI 味”变成固定替换词表。低质量文本问题常常来自结构、场景、信息密度和角色动机，而不只是词语。
- 不应自动把一次修改沉淀为长期风格规则。风格规则必须来自已发布正文与用户确认。

`ai-novel` 落地建议：

- 增加“AI 味检查”作为 reviewer 的子维度，报告中区分文字问题、结构问题和上下文问题。
- 从 AI artifact 与用户最终发布稿的 diff 中生成风格差异 proposal。
- 风格 proposal 必须由用户确认后才进入作品级风格规则，并保留来源 diff。

## 3. arboris 可借鉴点

### 3.1 面向作者的产品入口

可借鉴点：

- 首页和写作台围绕作者常见问题展开：角色是谁、故事在哪里、下一章写什么。
- 功能说明偏用户语言，而不是模型、队列、任务编号等工程细节。
- 设定管理、大纲、写作、版本对比形成清晰主路径。

不建议照搬：

- 不应把 `ai-novel` 做成单纯云端账号系统。当前定位仍应是本地优先、作品隐私优先。
- 不应为了界面简化隐藏发布安全链路。diff、backup、publish decision 对长篇正文安全很关键。

`ai-novel` 落地建议：

- 首页增加“继续写作”“待确认提案”“待处理 AI 草稿”“最近发布章节”四类作者任务。
- 写作页优先展示正文编辑、版本、diff、发布确认；队列、model call、raw JSON 放入高级区。
- 设置页提供“模型可用性检查”和“工作区健康检查”，但不展示密钥明文。

### 3.2 概念对话与蓝图

可借鉴点：

- 用概念对话帮助用户把零散想法整理成结构化蓝图。
- 蓝图可以编辑和局部覆盖，避免“一次生成就定稿”。
- 章节生成依赖蓝图、上一章桥接、摘要和检索结果，形成持续迭代闭环。

不建议照搬：

- 不应把蓝图作为单一巨大 JSON 取代文件工作区。`ai-novel` 已经以作品文件和 runtime 审计链为核心。
- 不应让 AI 自动覆盖用户蓝图。蓝图、设定和章纲都应 proposal-first。

`ai-novel` 落地建议：

- 在作品档案之上增加“创作问询”流程，用来补齐题材、目标读者、核心卖点、主线冲突、字数目标等字段。
- 蓝图输出映射为设定 proposal、总纲 proposal、卷纲 proposal 和章纲 proposal，而不是直接写源文件。
- 支持局部采纳：用户可只采纳角色设定、某一卷纲或某几章章纲。

### 3.3 多版本候选与评审

可借鉴点：

- 同一章节可以生成多个候选版本，再由用户选择或组合。
- 章节评审对候选版本提出改进建议，而不是直接替用户定稿。
- 手动编辑和版本选择都会触发摘要更新，为下一章提供真实上下文。

不建议照搬：

- 多版本不应默认增加成本。需要给用户明确选择：单候选稳定模式、多候选质量模式。
- 不应让“选择候选”绕过 `ai-novel` 的发布门。候选被选中后仍应进入版本、diff、backup 和发布确认。

`ai-novel` 落地建议：

- AI 工作台支持 1-N 个正文候选，并展示候选差异、review 结果和推荐原因。
- 候选进入“保存正文版本”后仍是未发布版本，必须经发布确认才写回正式正文。
- 发布后生成章节真实摘要，并作为后续上下文和 memory proposal 的输入。

### 3.4 RAG 与摘要闭环

可借鉴点：

- 已确认或已编辑章节进入摘要和向量索引，后续写作通过检索召回相关片段。
- 摘要和正文分块有生命周期：更新时覆盖旧索引，删除章节时清理过期向量。
- 检索失败时可以降级到蓝图和历史摘要，不阻断全部流程。

不建议照搬：

- 不应在早期强依赖复杂向量库。`ai-novel` 可以先用文件摘要、章节索引和轻量检索建立闭环。
- 不应让未发布草稿进入长期检索库，避免草稿污染后续章节。

`ai-novel` 落地建议：

- 先实现“已发布章节摘要 -> 最近章节摘要 -> 相关设定/角色/伏笔”的轻量上下文闭环。
- 后续再引入向量检索，并把向量来源限制为已发布正文和已确认资料。
- context report 必须标记每条上下文来自正式资料、已发布正文、proposal 还是 artifact；默认不读取未确认内容。

### 3.5 部署与使用体验

可借鉴点：

- README 面向用户说明 Docker、环境变量、常见问题和故障原因。
- 支持一体化部署，降低非开发者试用门槛。
- 常见错误说明覆盖模型未配置、超时、JSON 解析失败、章节纲要缺失等高频问题。

不建议照搬：

- 不应把部署说明和本地开发说明混在一个很长的使用路径里。普通作者和开发者需要分开入口。
- 不应承诺“开箱即全自动创作”。当前更稳妥的表达是“AI 辅助生产 + 人工确认发布”。

`ai-novel` 落地建议：

- 用户手册增加“普通作者启动路径”和“开发者启动路径”分栏。
- 运维手册补充常见 AI 错误、工作区扫描错误、发布门失败、备份恢复和模型预算暂停处理。
- 后续增加 Docker 或桌面启动器时，继续保持本地数据目录、工作区目录和仓库源码目录分离。

## 4. 两个参考项目对 ai-novel 的综合启发

### 可借鉴点

- 用明确创作阶段降低作者认知负担。
- 用作品文件系统承载长期上下文，不依赖对话记忆。
- 用写作卡锁定单章目标，降低正文生成漂移。
- 用多角色审查提升质量，但不让角色越权写回。
- 用发布门、备份和 diff 保护正式正文。
- 用摘要和检索建立长篇上下文闭环。
- 用用户语言组织界面，而不是暴露过多工程细节。

### 不建议照搬

- 不照搬参考项目的完整 prompt、skill 包或目录内容。
- 不把某个参考项目的交互方式变成唯一入口。
- 不把云端账号、复杂部署或外部 Agent 作为核心前提。
- 不把未确认 AI 输出、对标原文或草稿纳入长期记忆。

### ai-novel 落地建议

- 保持当前安全链路，把参考项目能力拆成 proposal、artifact、review、memory、style 等现有概念。
- 先做可验证的小闭环：作品档案补全、写作卡、正文候选、review、diff、发布、摘要更新。
- 再扩展对标资产、风格学习、RAG、批量流水线和桌面化体验。

## 5. 下一步

1. 先把“写作卡”和“作品档案补全”作为 P0，形成从设定/章纲到正文候选的稳定输入。
2. 增加“导入已有作品”的只读分析流程，先输出 proposal，不自动改正式资料。
3. 将 reviewer 报告统一为结构化 findings，便于 fixer、UI 和流水线消费。
4. 给发布后摘要和角色状态更新建立独立任务，让后续章节能稳定读取已确认上下文。
5. 将多版本候选作为质量优先模式，不作为默认生成方式。
