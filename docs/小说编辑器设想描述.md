# 小说编辑器设想描述

本文描述“小说编辑器”的目标形态，用于统一产品方向、前端信息架构、AI 流水线边界和后续迭代判断标准。它不是接口文档，也不是当前实现清单，而是后续开发应持续对齐的产品设想。

## 1. 一句话定位

小说编辑器是一个本地运行的长篇小说创作工作台：以人工写作为主，以 AI 自动生成和审核流水线为辅，帮助作者管理设定、章纲、正文、批注、正文版本、记忆和模型任务。

## 2. 目标用户

目标用户是有基础电脑操作能力、但不想理解工程术语的小说作者。

他们关心的是：

- 能不能稳定打开作品；
- 能不能舒服地写正文；
- 能不能快速查设定、章纲和前文；
- 能不能保存正文版本并找回历史内容；
- AI 生成的内容是否安全、可控、不会误覆盖正文；
- 自动流水线能否减少重复劳动，而不是增加操作负担。

## 3. 核心产品原则

1. 写作优先：正文区域必须大、干净、少遮挡。
2. 人工主控：作者手写内容不强制走 AI 检查。
3. AI 辅助：AI 生成、修订、审核必须作为独立辅助流程存在。
4. 版本优先：人工修改进入正文版本，模型输出进入草稿或提案，不直接覆盖源文件。
5. 发布受控：正文写回只能通过发布门，必须有校验、diff、备份和人工确认。
6. 设定谨慎：系统设定、小说设定、章纲默认只生成提案，不走普通直接发布。
7. 长篇友好：记忆系统按需注入上下文，不把全书全文塞给模型。
8. 可恢复：任务、草稿、审核、diff、备份、模型调用记录都应可追踪。

## 4. 两条主线

### 4.1 人工写作主线

这是默认主线，也是前端体验的第一优先级。

```text
打开作品
  -> 扫描素材
  -> 进入写作
  -> 打开章节
  -> 编辑正文
  -> 保存正文版本
  -> 查看版本历史
  -> 查看改动
  -> 确认发布版本
```

要求：

- 写作界面不能被模型任务、成本统计、审核 JSON 等内容打扰。
- 批注、版本、记忆可以在右侧栏查看，但右侧栏应可隐藏。
- 搜索、跳章、目录展开、全屏写作必须自然可用。
- 保存正文版本要像普通编辑器一样直观，但底层仍要保留校验、diff、备份和人工确认。

### 4.2 自动流水线辅线

自动流水线用于批量生成、批量检查和批量修订，适合沙盒验证或作者明确授权的批处理。

```text
选择章节范围
  -> 选择流水线模式
  -> 构建上下文
  -> AI 生成草稿
  -> AI 检查草稿
  -> AI 修订可修问题
  -> 复审
  -> 生成报告
  -> 人工确认是否写回
```

要求：

- 默认 dry-run，不覆盖真实正文。
- 任何失败都要有明确原因，例如模型失败、预算暂停、证据不足、需人工判断。
- 只有数据证明某模型岗位不适配时，才调整模型路由。
- 自动流水线不能绕过发布门。

## 5. 目标界面结构

### 5.1 首页

首页是独立入口，不是功能堆叠页。

应展示：

- 当前作品；
- 素材扫描状态；
- 章节数量；
- 最近打开章节；
- 待处理事项；
- 写作、AI 素材库、AI 工作台、自动流水线、设置入口。

首页的作用是帮助用户选择下一步，而不是直接完成复杂操作。

### 5.2 写作

写作界面应接近成熟编辑器的体验。

结构：

```text
左侧目录树
  - 系统设定
  - 小说设定
  - 章纲
  - 正文分卷

中间正文区
  - 搜索和跳转
  - 编辑正文
  - 保存正文版本
  - 全屏写作
  - 右键批注

右侧栏
  - 批注
  - 版本
  - 记忆
```

明确不放：

- 模型探测；
- 成本统计；
- 发布门高级信息；
- 大段审核 JSON；
- 自动流水线配置。

### 5.3 AI 素材库

AI 素材库用于阅读设定和章纲，并为 AI 上下文或人工采纳提供提案。

应支持：

- 系统设定、小说设定和章纲的只读查看；
- 提案生成；
- 提案 diff；
- 人工采纳说明。

默认不支持：

- 正文目录浏览；
- AI 直接覆盖设定源文件；
- 章纲被普通正文发布门写回。

### 5.4 AI 工作台

AI 工作台承载与草稿相关的 AI 操作。

应支持：

- 最近草稿列表；
- 当前章节上下文预览；
- 检查草稿；
- 按批注修订；
- 查看改动；
- 写回确认；
- 最近任务；
- 失败原因。

文案应面向作者：

- “检查草稿”优先于 “review artifact”；
- “查看改动”优先于 “diff”；
- “确认写回正文”优先于 “publish”；
- “额度已暂停”优先于 “paused_budget”。

工程字段应进入“高级详情 / 排错信息”，主路径不依赖手动输入草稿编号。

### 5.5 自动流水线

自动流水线是批处理控制台，但仍应使用作者能理解的表达。

应支持：

- 选择章节范围；
- 选择模式；
- 设置分批数量；
- 设置修订轮次；
- dry-run 开关；
- 创建、暂停、恢复、重试、停止；
- 查看每章状态；
- 生成运行报告。

状态表达：

- 已生成草稿；
- 已检查；
- 需修改；
- 需人工判断；
- 额度暂停；
- 已完成。

### 5.6 设置/模型

设置/模型用于高级信息和运维能力。

应支持：

- 工作区列表；
- 最近打开；
- 工作区扫描；
- 记忆库整理；
- 模型路由；
- 连通测试；
- 调用统计；
- 成本提示；
- 质量趋势；
- 上下文裁剪记录；
- skills 状态；
- 运行事件。

这些信息不应进入写作主界面。provider、model、token、base_url、原始 JSON 和产物 ID 应分层收进高级详情。

## 6. 视觉设想

整体应是“高级、安静、长期使用不疲劳”的创作工具，而不是营销页。

推荐方向：

- 写作区采用低干扰编辑器风格；
- 首页可有更强视觉设计，用作工作台入口；
- 可加入轻量动漫元素，但不能影响可读性和效率；
- 深色和浅色主题都必须可用；
- 暗色主题不能出现对比不足、文字发灰、按钮不可见的问题；
- 页面比例应稳定，状态浮层不能挤压正文。

借鉴方向：

- VSCode / Cursor：活动区、目录、编辑区、右侧栏、状态栏。
- 数据开发平台：清晰目录、顶部标签、右键操作、可隐藏侧栏。
- 文档类产品：干净版心、稳定滚动、清晰层级。

## 7. 数据与安全设想

仓库和作品应分离。

```text
系统代码仓库
  backend/
  frontend/
  config/
  docs/
  tests/

小说工作区
  00-系统/
  01-设定/
  02-正文/
  03-章纲/
  runtime/
```

运行态进入工作区 `runtime/`：

- artifacts：AI 草稿和提案；
- backups：写回前备份；
- diffs：改动对比；
- logs：模型调用和任务日志；
- reports：流水线和质量报告。

禁止：

- key、env、真实正文、runtime 进入 Git；
- 模型直接写正文；
- 无备份写回；
- 无人工确认写回。

## 8. AI 岗位设想

AI 应是强约束岗位，不是自由代理。

| 岗位 | 责任 | 禁止 |
| --- | --- | --- |
| writer | 生成正文草稿 | 判断是否通过审核、直接写回正文 |
| reviewer | 证据约束审核 | 修正文、扩写正文 |
| fixer | 按审核问题修订草稿 | 擅自重写无关段落 |
| memory | 整理短记忆和上下文 | 创作正文 |
| outliner | 生成章纲或结构提案 | 直接覆盖章纲源文件 |
| arbiter | 高风险判断辅助 | 绕过人工确认 |

审核原则：

- 有证据的问题才能进入自动修复；
- 无证据问题进入人工判断；
- 设定冲突、章纲矛盾、世界观缺口优先交给人工处理。

## 9. 记忆系统设想

记忆系统应像“上下文编译器”，不是摘要堆。

目标产物：

- 核心事实；
- 人物卡；
- 章节卡；
- 时间线；
- 伏笔表；
- 最近章节滚动摘要；
- 批注沉淀规则。

给模型时按需选择：

```text
当前章纲
+ 相关角色卡
+ 最近章节卡
+ 相关伏笔
+ 核心事实
+ 禁止违反项
```

如果超出预算，应记录 `context_degraded`，说明被裁剪的片段和原因。

## 10. 成功标准

一个版本是否接近目标形态，可以用以下标准判断：

1. 新用户能在 10 分钟内完成：打开作品、扫描、打开章节、编辑、保存正文版本、查看改动。
2. 写作页没有遮挡正文的候选、审核或发布面板。
3. 右键批注、搜索跳转、章节跳转、全屏写作可用。
4. 人工正文版本不被强制 AI 审核。
5. AI 草稿必须检查通过才能写回。
6. 设定和章纲不会被普通发布流程覆盖。
7. 自动流水线 dry-run 可生成报告，失败原因明确。
8. 模型质量和成本有数据，不凭感觉改路由。
9. 所有写回都有备份、diff、版本记录。
10. 前端主要文案使用作者语言，而不是工程术语。

## 11. 后续迭代方向

近期优先：

- 稳定写作页核心交互；
- 完善作品切换和目录体验；
- 增强正文版本历史；
- 让 AI 工作台流程更短、更清楚；
- 自动流水线先沙盒验证，再进入真实 dry-run。

中期优先：

- 多章节标签和跨章搜索；
- 更清晰的记忆可视化；
- 质量报告驱动模型路由；
- 章节级版本回滚；
- 批注规则沉淀为可编辑规则库。

长期设想：

- 插件化 skills；
- 更成熟的自动流水线编排；
- 多作品管理；
- 移动端只读审稿；
- 可选的云同步或备份，但本地安全边界优先。
