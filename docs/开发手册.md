# 小说编辑器开发手册

## 核心原则

- 代码仓库和小说工作区必须保持边界清晰。
- 模型输出先进入 artifact。
- 正文写回只允许发布门执行。
- 设定和章纲默认只生成提案，不自动覆盖。
- 作品档案、长期记忆补充和风格规则也必须 proposal-first，用户确认后才进入正式资料。
- 产品是通用长篇小说编辑器，不得把某个作品的设定、角色、题材或章节状态写入全局代码、全局 skill 或仓库级规则。
- reviewer 只审核，不修文、不写文。
- 无证据问题必须转 `admin` 或人工处理。
- 不读取、不打印、不提交 `key.txt`、`.env` 或真实 API key。

## 后端分层

```text
backend/app/api/       # FastAPI 路由
backend/app/services/  # 业务服务
backend/app/services/pipeline/ # 自动流水线
backend/app/db/        # SQLAlchemy 模型与会话
backend/tools/         # 本机工具和验收脚本
```

工作流应保持编排职责，不把 prompt、路径细节、发布细节全部塞进状态机。

AI 主生产流应保持单章粒度：批量任务可以排队多个单章子任务，但不能合并跨章节 artifact，也不能默认真实批量发布。

分层约束：

- `backend/app/core/file_utils.py` 只抛 `OSError` 或其子类，API 层通过全局 handler 转 HTTP 响应。
- 预算暂停使用 `ModelBudgetPausedError` 这类明确异常类型，不用 `"budget" in message` 做控制判断。
- 流水线内部状态推进使用返回值、状态机和 payload/result 记录，不用 `RuntimeError` 表示“准备好了，下一轮继续”。
- `/api/admin/*` 默认只允许本机访问；对外暴露服务时必须设置 `ADMIN_API_TOKEN`，前端用 `VITE_ADMIN_API_TOKEN` 透传。

## 前端分层

```text
frontend/src/pages/       # 页面
frontend/src/components/  # 组件
frontend/src/store*.ts    # Zustand 状态和持久化
frontend/e2e/             # Playwright 测试
```

写作界面优先保证正文区域无遮挡、右键批注可用、目录清晰、侧栏可隐藏。模型调用、任务、成本统计应和正文编辑分离。

## 发布门规则

发布正文前必须满足：

1. artifact 存在且 hash 匹配。
2. artifact 绑定章节和源文件。
3. AI 草稿必须存在最新 review，且 review 通过。
4. 人工草稿可跳过 AI review，但 artifact metadata 必须标记 `source=manual_editor_draft`、`requires_ai_review=false`，且仍要通过本地格式、标题、hash、diff 和人工确认校验。
5. review 的 candidate/source/version hash 与 artifact 一致。
6. source kind 是 `chapters`。
7. 人工确认发布。
8. 写入前生成 backup 和 diff。
9. 写入后重新扫描并更新短记忆。

强制发布只能放宽 AI review 结论，不能绕过 artifact 绑定、hash 校验、diff、backup、人工确认、publish decision 和事件审计。

## Proposal 与 Canonical 边界

以下内容必须先进入 proposal，用户确认后才允许写入正式资料：

- 作品档案补全
- 设定、大纲、卷纲、逐章章纲
- 单章写作卡
- 长期记忆补充
- 作品级风格规则

自动重建的短记忆只能来自已确认资料和已发布正文。未发布 artifact 不得进入长期记忆，也不得参与风格学习。

## 写作卡与生成模式

正文生成应优先使用单章写作卡，而不是把全文大纲、长样例或整本书正文塞进 prompt。

写作卡至少包含：章节编号、标题、本章目标、场景、出场人物、冲突/机制、常规推进方式、主角行动、情绪点、配角反应、结尾钩子、禁写项、字数范围和风格摘要。

生成模式应显式记录到 artifact metadata 或关联记录：

- `stable`：稳定省钱，默认模式，短 skill、低随机性、单候选。
- `quality`：质量优先，增加一致性检查、复审或 arbiter 辅助。
- `fast`：速度优先，减少 AI 审核，主要依赖人工检查。

context report 必须尽量记录写作卡、skill、memory、上下文裁剪和 generation mode，便于复盘 token 成本与输出稳定性。

仓库 `skills/` 只放通用规则。作品级风格、作品级审核清单和长期偏好放在对应小说工作区的 `skills/` 目录。模型上下文可以引用这些作品级规则，但不能把它们提交进系统仓库。

## 风格差异学习

风格学习来源是 AI 初稿和用户确认稿的 diff，以及用户风格类批注。它的输出是 style proposal，不是直接修改全局 skill。

限制：

- 不从未发布草稿学习。
- 不复制长段正文进 skill。
- 风格规则按作品隔离。
- 作品级 skill 只写入小说工作区，不写入仓库级 `skills/`。
- 每次更新保留来源 diff、版本和用户确认记录。

## 模型岗位

默认岗位：

- `writer` / `reviewer` / `fixer` / `quick_fix` / `memory` / `long_context` / `outliner` / `structural_fix` / `arbiter`：Agnes AI `agnes-2.0-flash`

DeepSeek、Kimi、Qwen、GLM 保留为可配置后备供应商。路由调整需要样本、成功率、证据率、复审通过率、耗时和成本数据支撑。

## 测试矩阵

常规提交前：

```powershell
python -m compileall -q .\backend .\tests
python -m pytest -q
cd .\frontend
npm run build
```

前端交互变化：

```powershell
cd .\frontend
npm run e2e
```

发布门或流水线变化：

```powershell
python -m backend.tools.sandbox_publish_smoke
python -m backend.tools.sandbox_pipeline_smoke --workspace runtime/sandbox_pipeline_workspace --chapters 3 --reset
```

模型治理变化：

```powershell
python -m backend.tools.probe_roles --roles writer reviewer fixer quick_fix memory long_context outliner structural_fix arbiter --key-file key.txt --force
python -m backend.tools.model_usage_report --out runtime\logs\model_usage_report.md
```

CI：

- `.github/workflows/ci.yml` 在 push / pull request 到 `main` 时运行后端编译、pytest、前端构建和 Playwright E2E。
- CI 只使用沙盒工作区，不依赖本机 `key.txt`、`.env`、真实小说目录或 runtime。

## 开源治理待办

高优先级已落地：I/O 异常分层、Admin API 基础保护、预算异常类型化、流水线 artifact 准备去异常控制流、`sha256_file` 分块读取、CI 基线。

后续分阶段推进：

1. 用 Pydantic model / dataclass 替换后端 services 中高频 `dict[str, Any]`，先从模型调用、流水线 run、artifact metadata 三处开始。
2. 将 `backend/app/services/pipeline/runs.py` 拆成 `runs_manager`、`runs_serializer`、`runs_report`，拆分前先补序列化快照测试。
3. 优化 `runs.py` 子任务状态统计，合并重复 SQL 查询。
4. 为前端补组件、hook、store 单元测试；页面继续增长时再引入路由库。
5. 将 5s 轮询改成按视图启停，并整理 `frontend/src/styles.css` 为更小的组件样式或 CSS Modules。
6. 添加 Dockerfile 和发布包说明，便于非开发用户部署。

P6 质量治理口径：

- 无样本时显示“数据不足”，不能把空样本解释为 0%。
- reviewer 关注证据率、无证据问题、manual_required 和 JSON 解析失败。
- writer 关注中文字符数是否落在 1900-2700 的硬容忍区间。
- fixer 关注修订后复审通过率，未复审只计入等待复审。
- skills 的“参与最近一次记录的上下文”来自 artifact 的 `context_report.skills`，没有记录时显示暂无记录。
- token/usage 只表示本地日志可见下限，真实消耗以供应商控制台为准。

百万字项目不得依赖全文 prompt。长期上下文应使用分卷摘要、最近章节摘要、人物状态、伏笔生命周期、时间线和相关设定检索。

## 打包边界

系统包只包含：

- `backend/`
- `frontend/`
- `config/`
- `content/` 空模板占位文件
- `skills/`
- `tests/`
- `docs/`
- 项目文档和依赖文件

系统包不包含：

- `00-系统/`
- `01-设定/`
- `02-正文/`
- `03-章纲/`
- `runtime/`
- `content/settings/`、`content/outlines/`、`content/chapters/` 下的真实素材
- 小说工作区内的作品级 `skills/`
- `key.txt`
- `.env`
- 缓存和构建产物
