# 通用长篇小说编辑器产品方案

## 1. 产品定位

`小说编辑器` 是一个本地运行的通用长篇小说 AI 辅助创作工作台。它不绑定单一作品，也不把某本书的设定、人物、题材或章节状态写入全局规则。

短期目标是支撑 20-30 万字小说的完整生产，覆盖修仙、玄幻、都市、科幻、悬疑、轻喜剧、情感等不同题材。长期目标是支撑百万字级别小说，通过分层记忆、人物状态、伏笔生命周期、时间线、审计记录和发布门，降低长篇创作中的设定漂移、人物跑偏、伏笔遗忘和 AI 味。

产品包含两条主路径：

- **AI 主导生产流**：AI 帮用户从构思推进到设定、大纲、章纲、单章写作卡、正文候选、审核、修复和风格归纳。
- **人工主导编辑流**：作者把它当作可靠的文档编辑器使用，AI 只作为扩写、润色、校验、一致性检查和风格学习助手。

## 2. 核心安全原则

所有长期资料和正文写回都必须遵守以下边界：

- AI 输出先进入 proposal 或 artifact，不直接覆盖源文件。
- 正文写回必须经过 review、diff、人工确认、backup、publish gate。
- 设定、大纲、章纲、长期记忆补充和风格规则均为 proposal-first，用户确认后才成为正式资料。
- 自动流水线默认 dry-run。批量流程只能排队多个单章任务，不能合并跨章节 artifact，也不能默认真实批量发布。
- 未发布草稿不进入长期记忆，也不参与风格学习。
- runtime 中的 artifact、review、diff、backup、model call、event、publish decision 默认保留；清理必须有范围、确认和事件记录。

## 3. 工作区与作品档案

每个作品应有独立工作区、独立 runtime、独立风格和独立记忆。支持三类目录结构：

```text
00-系统/ 01-设定/ 02-正文/ 03-章纲/
00-设定/ 01-大纲/ 02-正文/ 03-章纲/
content/settings/ content/outlines/ content/chapters/
```

建议每个作品维护一份作品档案，例如：

```text
00-设定/作品档案.md
```

作品档案字段：

- 书名
- 题材
- 目标字数
- 目标读者
- 叙事人称
- 整体基调
- 核心卖点
- 主角驱动
- 主要角色
- 世界规则摘要
- 禁写项
- 卷/章规划
- 当前生产阶段

作品档案按 settings 扫描。AI 可以生成补全 proposal，但用户确认前不能覆盖正式作品档案。

## 4. AI 主导生产流

标准流程：

```text
作品档案
-> 设定 proposal
-> 大纲 proposal
-> 卷纲 proposal
-> 逐章章纲 proposal
-> 单章写作卡
-> 正文 artifact
-> reviewer 审核
-> fixer 修复
-> 复审
-> diff
-> 人工确认发布
-> 记忆/风格 proposal
```

流程规则：

- 缺失信息通过结构化弹窗问询，用户也可以跳过并让 AI 生成建议 proposal。
- 正文最小生产单位是单章。
- writer 只生成正文候选，不判断是否通过。
- reviewer 只审核，不改写正文。
- fixer 只修复 reviewer 授权且证据充分的问题。
- arbiter 只做高风险判断辅助，不绕过发布门。
- 发布成功后重新扫描并更新短记忆。

## 5. 单章写作卡

为降低“每次抽奖”和 token 浪费，正文生成前先锁定单章写作卡。writer 优先读取写作卡、必要设定摘要、最近记忆和短生产 skill，而不是读取全文大纲或长样例。

写作卡字段：

- 章节编号
- 章节标题
- 本章目标
- 场景
- 出场人物
- 冲突/机制
- 常规解法或常规推进
- 主角行动
- 情绪点/爽点/悬念点
- 配角反应
- 结尾钩子
- 禁写项
- 字数范围
- 风格摘要

写作卡本身先作为 proposal/artifact 保存。用户确认后，它才能作为稳定正文生成输入。context report 必须记录写作卡、skill、memory 是否参与上下文。

## 6. 生成模式

正文和 AI 辅助编辑应支持三种模式：

- **稳定省钱**：默认模式。短 skill、低随机性、单候选、常规 reviewer，优先减少漂移和 token 消耗。
- **质量优先**：增加一致性检查、复审或 arbiter 辅助，适合关键章节。
- **速度优先**：减少 AI 审核步骤，主要靠人工审核推进，适合低风险草稿。

生成模式应写入 artifact metadata 或 model call 关联记录。稳定模式不应默认 force 重新生成；相同输入允许复用缓存。

## 7. 风格差异学习

风格学习的目标是减少 AI 味，而不是把所有草稿都变成长期规则。

输入：

- AI 初稿 artifact
- 用户最终确认稿
- diff
- 用户风格类批注

输出：

- 风格差异报告
- AI 味问题清单
- 用户偏好表达清单
- style proposal
- 用户确认后的作品级风格规则

限制：

- 未发布草稿不参与学习。
- 不复制长段正文进 skill。
- 不自动更新全局风格。
- 风格规则按作品隔离。
- 每次风格更新保留来源 diff 和版本，可回滚。

## 8. 人工主导编辑流

写作页首先是文档编辑器，不应依赖模型可用性。

基础能力：

- 新建章节
- 保存正文版本
- 删除、软删除和恢复
- 复制章节
- 搜索
- 批注
- 版本历史
- diff
- 发布确认
- 备份提示
- 可隐藏侧栏

AI 辅助能力：

- 扩写
- 润色
- 降低 AI 味
- 错字校验
- 语病校验
- 格式校验
- 一致性检查
- 逻辑 bug 检查
- 伏笔检查
- 人物口吻检查

所有 AI 编辑结果先生成候选。用户将候选应用到编辑稿后，它仍只是未发布版本，不能直接成为正式正文。

## 9. 百万字扩展方向

百万字小说不能依赖全文塞入 prompt，应使用分层摘要和检索式上下文。

推荐上下文：

```text
当前写作卡
+ 当前卷摘要
+ 最近 3-5 章摘要
+ 相关人物卡
+ 相关设定条目
+ 相关伏笔
+ 短风格规则
```

长期模块：

- 分卷摘要
- 人物状态卡
- 伏笔生命周期：埋下、推进、回收、废弃
- 时间线
- 地点和组织关系
- 设定变更影响分析
- 重复桥段检测
- 角色口吻漂移检测
- 卷末复盘和下一卷约束

长期验收指标应包含章节数量、扫描耗时、上下文构建耗时、token 成本、记忆命中率和发布失败恢复能力。

## 10. 落地顺序

第一批落地只处理低风险基础：

1. 文档和边界修正：通用产品定位、三种工作区结构、安全链路、强制发布定义。
2. 作品档案：模板、扫描、补全 proposal、前端入口。
3. 单章写作卡：由章纲生成、用户确认、writer 使用、context report 记录。
4. 生成模式：稳定省钱默认，质量优先和速度优先可选。
5. 风格差异学习：从 AI 初稿和用户确认稿 diff 中生成 style proposal。
6. 人工编辑器补齐：新建、复制、软删除/恢复、选区 AI 候选。
7. 百万字扩展：先文档化指标和资料分层，再逐步实现。

## 11. 验收场景

- 新建非当前题材作品时，不加载任何单一作品规则。
- 作品档案缺字段时弹窗问询。
- 设定、大纲、章纲 proposal 未确认前不入库。
- 单章写作卡确认后才能作为稳定正文生成输入。
- AI 草稿未 review 通过不能发布。
- 强制发布也必须保留理由、diff、backup、publish decision。
- AI 初稿和用户修改稿可生成风格差异报告。
- 风格规则未确认前不影响后续生成。
- 人工编辑器在模型不可用时仍能写作、保存版本、发布手写版本。
- 批量流水线只排队单章任务，不真实批量发布。
- 百万字模拟上下文不塞全文，只用分层摘要和检索条目。
