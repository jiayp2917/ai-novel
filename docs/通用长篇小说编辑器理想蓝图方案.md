# 通用长篇小说编辑器理想蓝图方案

## 1. 蓝图定位

本文档描述 `小说编辑器` 的理想终局形态，用于指导后续产品、工程、测试和内容生产决策。它不同于短期实施计划：实施计划回答“下一步先做什么”，蓝图回答“最终应该长成什么样，以及哪些边界不能被破坏”。

产品目标是成为一个本地优先、人工可控、AI 辅助的通用长篇小说生产工作台。短期支撑 20-30 万字作品跑通从构思到发布的闭环，长期支撑百万字级别作品的分层记忆、人物状态、伏笔管理、风格学习、成本控制和发布审计。

产品不绑定某一本小说，也不把任何单一作品的人物、题材、世界观或章节状态写入全局规则。具体作品信息只存在于对应工作区、作品级配置、作品级 skill 或明确的计划文件中。

## 2. 设计原则

1. **作者拥有最终控制权**
   - AI 负责生成候选、提案、检查和修复建议。
   - 作者负责确认设定、确认章纲、确认写作卡、确认正文发布。
   - 任意 AI 输出不得默认成为正式资料或正式正文。

2. **正文写回必须可追踪**
   - 正文写回必须经过 artifact、review、diff、人工确认、backup、publish gate。
   - 发布后必须能追踪来源 artifact、审核结果、差异、确认人、发布时间和备份位置。

3. **长期资料 proposal-first**
   - 设定、大纲、卷纲、章纲、作品档案、记忆补充、风格规则都先进入 proposal。
   - proposal 未确认前只能作为候选参考，不能进入 canonical 上下文。

4. **单章是最小生产单位**
   - 正文生成、审核、修复、发布、回滚和统计都以单章为核心单位。
   - 批量流程只能排队多个单章任务，不能合并跨章节 artifact，更不能默认批量真实发布。

5. **上下文稳定可复现**
   - writer 生成正文时优先读取单章写作卡、必要设定摘要、近期记忆、相关人物状态和短 skill。
   - 不把全文大纲、长样例或整本正文直接塞进 prompt。
   - artifact metadata 必须记录生成模式、写作卡、skill、memory 和关键上下文来源。

6. **不同作品彻底隔离**
   - 工作区、runtime、作品档案、记忆、风格、模型调用统计和发布记录按作品隔离。
   - 新建非当前题材作品时，不应加载任何旧作品专属规则。

7. **安全默认**
   - 自动流水线默认 dry-run。
   - API key、`.env`、`key.txt` 不读取、不展示、不提交。
   - runtime 审计链路默认保留，清理必须有范围、确认和事件记录。

## 3. 目标用户与核心场景

### 3.1 目标用户

- 需要长期稳定产出的网文作者。
- 有明确构思，但希望 AI 辅助补设定、拆章纲、生成初稿和做一致性检查的作者。
- 喜欢手写正文，但需要 AI 做扩写、润色、校验、伏笔检查和风格归纳的作者。
- 需要管理多题材、多作品、多版本内容的创作者。

### 3.2 核心场景

1. **AI 主导生产**
   - 用户输入题材、卖点、人物、基调和目标字数。
   - AI 补全作品档案、设定、大纲、卷纲和逐章章纲。
   - 用户逐步确认后，AI 基于单章写作卡生成正文候选。
   - 正文经过审核、修复、diff 和人工确认后写回。

2. **人工主导编辑**
   - 用户把应用当作小说文档编辑器。
   - 支持新建、保存、复制、删除、搜索、批注、版本历史和 diff。
   - AI 仅作为选区扩写、润色、错字校验、语病校验、格式校验、一致性检查和逻辑 bug 检查助手。

3. **长篇维护**
   - 发布后自动更新短记忆、人物状态、伏笔状态和卷摘要 proposal。
   - 用户确认后进入长期资料。
   - 后续章节使用分层检索，而不是依赖一次性长 prompt。

## 4. 信息架构

前端保持六个清晰入口：

1. **首页**
   - 当前作品状态、继续写作、待处理事项、最近章节、风险提醒。
   - 不展示过多模型字段、任务编号或工程细节。

2. **写作**
   - 以正文编辑为中心。
   - 目录、批注、版本、记忆和 diff 可隐藏。
   - 支持人工写作、保存版本、查看改动、确认发布。

3. **AI 素材库**
   - 查看设定、大纲、卷纲、章纲、作品档案和提案。
   - 支持生成 proposal、查看差异和确认采用。
   - 不展示正文发布按钮。

4. **AI 工作台**
   - 主路径：选择草稿、检查、查看改动、确认写回。
   - 编号、队列、JSON、模型原始详情进入高级区。

5. **自动流水线**
   - 默认只生成草稿，不写回正文。
   - 展示每章状态、当前步骤、失败原因、下一步动作和报告位置。
   - pause、resume、cancel、retry 只在合法状态出现。

6. **设置/模型**
   - 主卡只展示模型用途、接口地址、密钥状态、测试连接、保存配置。
   - provider、token、api_key_env、raw JSON、调用记录和日志默认折叠到高级排错区。

## 5. 交付形态蓝图

面向无代码能力的小说作者，产品最终应以本地安装包为主要交付形态。GitHub 只作为源码、版本发布和开发协作入口，不作为普通作者的使用入口。

### 5.1 阶段 A：浏览器访问型安装包

第一阶段优先做“点击启动，自动打开本机网页”的安装包，尽量复用当前 React + FastAPI 架构。

目标体验：

```text
用户安装 小说编辑器
-> 点击桌面图标
-> launcher 启动本机 FastAPI 后端
-> 初始化/迁移 SQLite 和用户数据目录
-> 托管或启动前端静态页面
-> 自动打开浏览器 http://127.0.0.1:15173
-> 用户在浏览器中完成写作、AI 生成、审核、diff 和发布
```

阶段 A 必须具备：

- Windows 安装包或免安装包。
- 启动器负责端口检测、后端启动、前端打开、进程退出和错误提示。
- 前端生产构建由本机服务托管，不要求用户运行 `npm`。
- Python 后端和依赖被打包，不要求用户安装 Python。
- 数据目录默认放在用户本机应用数据目录，而不是源码目录。
- 首次启动向导支持选择或创建作品工作区。
- 设置页支持保存模型密钥状态，不要求用户手写 `.env` 或 `key.txt`。
- 模型不可用时，人工写作、保存版本、diff 和发布手写版本仍可用。

阶段 A 的目标不是“像原生软件”，而是让普通作者无需命令行即可真实使用和反馈。

### 5.2 阶段 B：桌面窗口型安装包

第二阶段在阶段 A 稳定后，再升级为 Electron 或 Tauri 桌面窗口形态。用户不需要感知本地网址，打开后看到的是完整桌面应用。

目标体验：

```text
用户安装 小说编辑器
-> 点击桌面图标
-> 桌面窗口打开
-> 后端随应用自动启动
-> 用户在应用窗口中操作
-> 退出应用时安全停止后台进程
```

阶段 B 增强能力：

- 独立桌面窗口。
- 应用菜单、文件选择、最近作品、日志导出。
- 自动更新或半自动更新。
- 崩溃恢复和后台进程守护。
- 更好的系统托盘、关闭确认和数据备份提示。
- 安装包签名，降低系统安全警告。

阶段 B 不改变核心安全链路：AI 输出仍先进入 proposal/artifact，正文写回仍必须经过 review、diff、人工确认、backup 和 publish gate。

### 5.3 不优先做纯云端 SaaS

当前阶段不把纯网页 SaaS 作为主路径。原因：

- 小说正文、设定和大纲具有强隐私属性。
- 用户 API key、模型成本和版权责任需要更复杂的云端治理。
- 长篇 runtime、artifact、backup、diff 和审计记录更适合本地优先。
- SaaS 会引入账号、计费、存储、合规和运维问题，容易分散当前对写作闭环的验证。

可以保留网页演示版用于收集 UI 和流程反馈，但演示版不得作为真实正文写作和发布的默认入口。

## 6. 核心数据模型

理想模型不要求一次性重构数据库，但后续功能应尽量向以下概念收敛。

| 概念 | 作用 | 写入规则 |
| --- | --- | --- |
| Workspace | 单个作品工作区 | 用户选择或配置 |
| WorkProfile | 作品档案 | proposal 确认后写入 |
| SourceFile | 设定、大纲、章纲、正文源文件索引 | 扫描生成 |
| SourceProposal | 设定、大纲、章纲、档案提案 | AI 生成，确认后采用 |
| Chapter | 可发布正文单元 | 扫描或用户创建 |
| ChapterVersion | 人工编辑版本或发布版本 | 保存版本生成 |
| WritingCard | 单章写作卡 | proposal 确认后作为 writer 输入 |
| Artifact | AI 正文候选或修订候选 | AI 生成，不能直接写源文件 |
| Review | 审核结果 | reviewer 生成 |
| Diff | 发布前差异 | 发布前必须生成 |
| PublishDecision | 发布确认记录 | 人工确认后生成 |
| Backup | 写回前备份 | publish gate 自动生成 |
| MemoryItem | 长期或短期记忆 | 由已确认资料和已发布正文生成 |
| StyleRule | 作品级风格规则 | style proposal 确认后生效 |
| ModelCall | 模型调用记录 | 自动记录，可高级清理 |
| Event | 审计事件 | 关键操作自动记录 |
| PipelineRun | 自动流水线任务 | 默认 dry-run |

## 7. AI 生产流蓝图

### 7.1 标准流程

```text
作品构思
-> 作品档案 proposal
-> 设定 proposal
-> 总纲 proposal
-> 卷纲 proposal
-> 逐章章纲 proposal
-> 单章写作卡 proposal
-> 用户确认写作卡
-> 正文 artifact
-> reviewer 审核
-> fixer 修复
-> 复审
-> diff
-> 人工确认
-> backup
-> publish gate 写回
-> 扫描
-> 记忆/风格 proposal
```

### 7.2 缺失信息处理

当作品档案、设定、大纲或写作卡缺少关键字段时，应优先弹出结构化问询。用户可以：

- 直接回答，系统写入 proposal。
- 选择跳过，让 AI 生成建议 proposal。
- 标记为暂不需要，后续再补。

系统不得为了推进流程而静默编造 canonical 资料。

### 7.3 生成模式

| 模式 | 目标 | 适用场景 | 默认行为 |
| --- | --- | --- | --- |
| 稳定省钱 | 输出稳定、成本低 | 日常章节 | 短 skill、低随机性、单候选、常规 reviewer |
| 质量优先 | 降低关键章节风险 | 开篇、高潮、转折、结尾 | 增加一致性检查、复审或 arbiter 辅助 |
| 速度优先 | 快速出草稿 | 低风险过渡章 | 减少 AI 审核，人工承担更多审核 |

生成模式必须进入 artifact metadata、model call 关联记录或 context report。

## 8. 单章写作卡蓝图

单章写作卡是稳定正文生产的核心输入。理想字段如下：

- 章节编号
- 章节标题
- 所属卷
- 本章目标
- 起始状态
- 结束状态
- 场景
- 出场人物
- 人物状态变化
- 冲突或机制
- 常规解法或常规推进
- 主角行动
- 情绪点、爽点或悬念点
- 配角反应
- 信息增量
- 伏笔埋设或回收
- 结尾钩子
- 禁写项
- 字数范围
- 风格摘要
- 依赖设定和依赖记忆

验收标准：

- 任意一张确认后的写作卡，应能直接交给 writer 生成 2000-3000 字章节候选。
- 写作卡必须足够具体，不能出现“自由发挥”“补一段日常”“写得精彩一点”这类空泛指令。
- 同一写作卡在稳定模式下生成的正文结构应接近，不应出现题材、人物动机或剧情目标漂移。

## 9. 上下文与记忆蓝图

### 9.1 上下文优先级

writer 生成正文时推荐上下文顺序：

```text
确认后的单章写作卡
-> 当前卷摘要
-> 最近 3-5 章摘要
-> 相关人物状态卡
-> 相关设定条目
-> 相关伏笔
-> 作品级短风格规则
-> 禁写项
```

不推荐默认读取：

- 全文原稿。
- 全量大纲。
- 长篇旧素材样例。
- 未确认 proposal。
- 未发布草稿。

### 9.2 记忆分层

| 层级 | 内容 | 更新来源 |
| --- | --- | --- |
| 作品摘要 | 全书核心设定、主线、基调 | 已确认作品档案和总纲 |
| 卷摘要 | 本卷目标、阶段变化、主要冲突 | 已确认卷纲和已发布章节 |
| 章节摘要 | 每章事实、人物变化、伏笔 | 已发布正文 |
| 人物状态卡 | 目标、关系、能力、口吻、当前状态 | 已发布正文和用户确认资料 |
| 伏笔卡 | 埋设、推进、回收、废弃 | 已发布正文和人工确认 |
| 风格规则 | 句式、对白、节奏、禁忌 | 用户确认的 style proposal |

未发布正文、未确认提案和失败草稿不能进入长期记忆。

## 10. 风格学习蓝图

风格学习不等于复制用户正文，也不等于把所有偏好写成永久规则。它应当从“AI 初稿”和“用户确认稿”的差异中提炼短规则。

流程：

```text
AI artifact
-> 用户修改并发布
-> diff
-> 风格差异报告
-> style proposal
-> 用户确认
-> 作品级风格规则版本
```

style proposal 应包含：

- AI 味问题清单。
- 用户偏好表达清单。
- 对白节奏规则。
- 叙述密度规则。
- 爽点、情绪点或悬念点表达规则。
- 禁止事项。
- 示例可以短句化，但不得复制长段正文。

风格规则必须按作品隔离、版本化、可回滚。

## 11. 发布门蓝图

### 11.1 人工正文发布

```text
编辑正文
-> 保存正文版本
-> 查看 diff
-> 确认发布
-> backup
-> hash 校验
-> 写回源文件
-> publish decision
-> 重新扫描
```

人工正文版本可以不经过 AI review，但仍必须查看 diff、备份、确认和审计。

### 11.2 AI 正文发布

```text
AI artifact
-> review
-> 必要时 fixer 修复
-> review 通过
-> diff
-> 人工确认
-> backup
-> publish gate
-> publish decision
```

禁止路径：

- AI 直接覆盖正文。
- review 未通过直接发布。
- 未生成 diff 发布。
- 未备份发布。
- 自动流水线默认真实写回。
- 缺少 `base_chapter_id`、`base_source_file_hash`、`base_chapter_version_id` 的 artifact 进入 publish。

## 12. 自动流水线蓝图

自动流水线定位为“辅助排队和观察”，不是自动出版系统。

理想能力：

- 创建 dry-run 任务。
- 展示每章状态和当前步骤。
- 展示失败原因、manual_required 原因和下一步动作。
- 支持合法状态下的 pause、resume、cancel、retry。
- 生成 runtime 报告。
- 保留 artifact、review、diff 和事件记录。

安全默认：

- 默认“只生成草稿，不写回正文”。
- 不提供普通用户一键批量真实发布入口。
- 真实发布必须回到单章发布门。

## 13. 人工编辑器蓝图

写作页应在模型不可用时仍然完整可用。

基础能力：

- 新建章节。
- 编辑正文。
- 保存版本。
- 复制章节。
- 软删除和恢复。
- 搜索。
- 批注。
- 版本历史。
- diff。
- 发布确认。
- 备份提示。
- 全屏写作。
- 可隐藏目录和辅助面板。

AI 选区能力：

- 扩写。
- 润色。
- 降低 AI 味。
- 错字校验。
- 语病校验。
- 格式校验。
- 一致性检查。
- 逻辑 bug 检查。
- 伏笔检查。
- 人物口吻检查。

所有 AI 选区结果先生成候选。应用到编辑稿后也只是未发布版本，不能直接成为正式正文。

## 14. 工程架构蓝图

### 14.1 后端模块边界

- workspace scanner：只负责识别文件和建立索引，不修改源文件。
- source proposal：负责设定、大纲、章纲、作品档案提案。
- writing card：负责从章纲生成单章写作卡 proposal。
- context builder：负责构建可追踪上下文报告。
- writer：只负责正文候选。
- reviewer：只负责审核。
- fixer：只修复 reviewer 授权问题。
- diff service：负责差异计算。
- publish gate：负责备份、hash 校验、写回和发布记录。
- memory service：只从已确认资料和已发布正文生成记忆。
- style service：只从发布 diff 生成风格 proposal。
- pipeline：负责排队和状态管理，默认 dry-run。

### 14.2 前端边界

- 首页：作品状态和待办。
- 写作：正文编辑和发布。
- AI 素材库：资料查看和 proposal。
- AI 工作台：草稿审核、修复和写回。
- 自动流水线：dry-run 任务观察。
- 设置/模型：模型配置、连接测试和高级排错。

前端不应把后端安全策略做成纯 UI 约束。关键发布门必须由后端再次校验。

### 14.3 安装包运行边界

阶段 A 和阶段 B 都应保持同一套后端安全校验。安装包只改变启动和交付方式，不改变业务安全语义。

安装包运行时应拆分：

- 应用安装目录：只放程序文件和前端静态资源。
- 用户数据目录：放 SQLite、runtime、日志、模型配置状态和应用配置。
- 作品工作区：放用户小说设定、章纲、正文和作品级 runtime。
- 临时目录：放启动锁、临时报告和崩溃转储。

启动器职责：

- 检查端口是否可用。
- 选择空闲本机端口或使用固定端口。
- 启动后端并等待 `/health` 可用。
- 自动打开浏览器或桌面窗口。
- 捕获后端启动失败原因并给出中文提示。
- 退出时安全停止子进程。
- 不读取或打印 API key 明文。

## 15. 可观测性蓝图

每次 AI 或发布操作都应能回答：

- 用了哪个模型角色和模型。
- 读取了哪些上下文。
- 是否使用写作卡。
- 是否使用 skill。
- 是否使用 memory。
- 生成模式是什么。
- 输出保存在哪里。
- 审核是否通过。
- diff 是什么。
- 谁确认发布。
- 备份在哪里。
- 发布后源文件 hash 是否匹配预期。

普通用户默认看到通俗状态，高级字段进入排错区。

## 16. 百万字扩展蓝图

百万字小说的核心不是扩大 prompt，而是降低每次生成所需上下文。

必须具备：

- 分卷摘要。
- 章节摘要。
- 人物状态卡。
- 伏笔生命周期。
- 时间线。
- 地点和组织关系。
- 设定变更影响分析。
- 重复桥段检测。
- 人物口吻漂移检测。
- 卷末复盘。
- token 成本统计。
- 上下文命中率统计。

百万字验收场景：

- 1000 章模拟数据下，素材扫描和上下文构建仍可用。
- 生成第 800 章时不会读取全书正文。
- 能检索到与当前写作卡相关的人物、伏笔、设定和最近章节摘要。
- 能指出未回收伏笔和人物状态冲突。

## 17. 分阶段落地路线

### P0：边界和文档

- 通用产品定位。
- 工作区结构说明。
- 安全发布门定义。
- 交付形态确定为先阶段 A 浏览器访问型安装包，再阶段 B 桌面窗口型安装包。
- 当前作品上下文移出全局规则。
- 代码和文档不再固化单一小说设定。

### P1：最短可用闭环

- 阶段 A 安装包方案设计：后端打包、前端托管、launcher、用户数据目录和端口策略。
- 作品档案模板。
- 作品档案补全 proposal。
- 单章写作卡 proposal。
- writer artifact 记录 generation_mode。
- sandbox 发布门验证。
- 前端能看见 proposal 和 artifact。

### P2：稳定正文生产

- 阶段 A 可用内测包：用户点击启动后自动打开本机网页前端。
- 确认后的写作卡进入 writer 上下文。
- context report 记录写作卡、skill、memory。
- AI 工作台完成检查、修复、diff、确认写回。
- 自动流水线 dry-run 可观察。

### P3：风格和记忆

- 已发布正文生成章节摘要 proposal。
- 人物状态和伏笔卡 proposal。
- AI 初稿与用户确认稿 diff 生成 style proposal。
- 作品级风格规则版本化。

### P4：人工编辑器补齐

- 新建、复制、软删除、恢复章节。
- 选区 AI 候选。
- 批注驱动修订。
- 全屏写作和可隐藏面板。

### P5：长篇扩展

- 阶段 B 桌面窗口型安装包：Electron 或 Tauri 桌面窗口、日志导出、更新机制和崩溃恢复。
- 百万字模拟测试。
- 分层检索优化。
- 成本和质量趋势。
- 重复桥段、口吻漂移和伏笔风险看板。

## 18. 验收清单

### 产品验收

- 阶段 A 中，普通作者安装后点击桌面图标即可打开本机网页前端，无需命令行。
- 新建不同题材作品时，不加载旧作品规则。
- 作品档案缺字段时能问询或生成 proposal。
- 设定、大纲、章纲未确认前不成为正式资料。
- 单章写作卡确认后才能作为稳定正文生成输入。
- AI 草稿必须先 artifact，再 review，再 diff，再人工确认发布。
- 人工手写版本也必须查看 diff、备份和确认。
- 自动流水线默认 dry-run。
- 模型不可用时仍可手写、保存版本和发布手写版本。

### 工程验收

- 阶段 A 安装包能启动后端、打开前端、初始化数据库、选择工作区，并在端口占用时给出中文提示。
- 阶段 A 不要求用户安装 Python、Node.js、npm 或手动运行迁移。
- 阶段 B 桌面窗口关闭时能安全停止后端子进程，不留下僵尸进程。
- `python -m compileall -q .\backend .\tests` 通过。
- `python -m pytest -q` 通过。
- `cd frontend && npm run build` 通过。
- 前端交互变更后 `npm run e2e` 通过或记录未运行原因。
- 发布门变更后 sandbox publish smoke 通过。
- pipeline 变更后 sandbox pipeline smoke 通过。

### 安全验收

- 安装包不把 API key 写入安装目录、日志或崩溃报告。
- 安装包升级不覆盖用户作品工作区、用户数据目录和 runtime 审计链路。
- 不读取、不展示、不提交 `key.txt`、`.env`、API key。
- 不直接改真实小说源文件。
- 不删除 runtime 审计链路。
- 不绕过 review、diff、backup、publish decision。
- 不允许缺少 base 绑定的 artifact 发布。

## 20. 后续决策规则

后续任何新功能都应先回答四个问题：

1. 是否会绕过 proposal 或 publish gate？
2. 是否会把单一作品规则写进全局？
3. 是否能在模型不可用时保持人工编辑主流程可用？
4. 是否能被测试、审计和回滚？

若任一答案不清楚，先做文档和 sandbox 验证，不直接进入真实作品写回。
